# AI Agent 実装規約

本番システムの一部として LLM を組み込むコード（tool calling、structured output、multi-turn workflow を伴うもの）に対する **必須** ルール。Claude Code 自体の運用や使い捨てスクリプトは対象外。

## 1. 基本方針

- 決定的な処理と非決定的な処理を明確に分離すること。決定的な処理はアプリケーションコードで実装し、非決定的な判断のみを LLM に委ねる。
- プロンプト、ツール定義、ワークフロー定義は本リポジトリ内でコードとしてバージョン管理し、差分追跡可能な状態にすること。
- LLM 呼び出しサイトは prompt + tool schema + model id を明示的・名前付きの入力として受け取ること。サービスメソッド内にプロンプトをハードコードしない。

## 2. テスト方針

決定的な処理には unit test を **必須** とする。少なくとも以下は unit test の対象とすること。

- 入出力のバリデーション
- JSON / schema の整形
- ツール呼び出しの分岐
- 権限判定
- リトライ、タイムアウト、フォールバック制御
- 状態管理およびワークフロー制御

必須事項:

- LLM の応答品質を unit test の対象としないこと。Unit test は決定的な挙動を検証するものであり、LLM の挙動は §3 の eval で検証する。
- Agent コードの unit test は本リポジトリの他コードと同じ規約に従う（`.claude/rules/testing.md` 参照）。

## 3. Eval 方針

新規 agent または既存 agent への主要な変更を加える場合、以下を **必須** とする。

- 評価用データセットをリポジトリにコミットすること。
- Eval ケースには少なくとも以下を含めること:
  - 正常系
  - 失敗しやすいケース
  - 境界ケース
  - セキュリティ上の危険ケース（prompt injection、ツール濫用、データ流出など）
- 評価指標は agent の目的に応じて、タスク完了率 / 形式準拠率 / ツール呼び出し成功率 / コスト / レイテンシ から選定すること。汎用セットを無自覚に流用しない。
- 以下のいずれかが変わった場合は eval を **再実行** すること: prompt、model id、tool schema。Eval 結果は変更（PR 説明文または成果物リンク）に添付する。

## 4. プロンプト管理

- system prompt、developer prompt、few-shot 例、tool schema はコードと同等のレビュー対象とすること。
- プロンプトはコードベース内に明示的に置くこと（例: `prompts/` ディレクトリ、`//go:embed` 経由）。本番経路でリモートのみのプロンプトソースは禁止する。
- 本番で使用中の prompt version を稼働システムから特定可能にすること（agent 実行時に prompt version をログ出力する。§6 参照）。

## 5. 実装原則

以下のタスクは LLM に委ねず、コードで実装すること（**必須**）。

- 計算
- 検索条件の厳密化（例: SQL フィルタ構築）
- データ整形 / 正規化
- 権限判定
- ユーザーや下流システムに返す最終出力の検証

ツール設計:

- ツールの入出力は schema で明示すること。フリーフォーム文字列のみのツールは禁止。
- ツールは可能な限り idempotent に設計すること。
- 破壊的またはその他の危険操作を行うツールは、明示的な確認手順または権限制御ゲートを **必須** とする。「LLM が呼ぶと判断した」は正当化にならない。

## 6. 可観測性

Agent 呼び出し時には少なくとも以下を記録可能にすること。

- 入力
- 使用モデル ID
- prompt version
- ツール呼び出し履歴（各呼び出しの name / args / result）
- 最終出力
- エラー内容
- 実行時間
- コスト（token 数、可能なら $ 換算）

必須事項:

- ロギングは `slog` 規約に従う（`.claude/rules/coding-standards.md` §6）。
- 機密情報を不必要にログしないこと。PII、認証情報、機密エンドポイントの完全リクエストボディはログハンドラに到達する前に redact する。

## 7. 失敗前提の設計

Agent 実装は失敗を前提に設計すること。少なくとも以下を考慮する。

- タイムアウト
- リトライ（上限あり）
- フォールバック
- 中断と再実行
- 人手介入への切り替え

必須事項:

- 無制限リトライ禁止。すべてのリトライ経路に明示的な max-attempts と backoff を設定する。
- ツール呼び出しサイクルでの無限ループ禁止。モデルがツール呼び出しを提案し続けた場合でも、step budget で確実に終了すること。
- 失敗経路は呼び出し元に型付きエラーとして表面化する。HTTP 経由 agent では `shared/utilhttp` のエラー型を用いる（`.claude/rules/coding-standards.md` §3）。

## 8. レビュー観点

通常のコードレビュー観点に加え、レビュアは以下を **必須** で確認すること。

- LLM とコードの責務分担が妥当であり、決定的な処理を LLM に外注していないか。
- §2 の各項目が unit test でカバーされているか。
- Eval が変更内容を十分にカバーしているか。Prompt のみの変更でも eval を再実行し、結果が添付されているか。
- 危険なツール権限が認可ゲートなしにモデルへ露出していないか。
- ログと再現性（prompt version、model id、入力）が確保されているか。

## 短縮版（最低ライン）

- AI agent の品質保証は `unit test + integration test + eval` を基本とする。
- 決定的な処理はコード化し、unit test で担保する。
- LLM の応答品質は eval で担保する。prompt / model / tool 変更時は eval を再実行する。
- プロンプトとツール定義はコードとしてバージョン管理し、レビュー対象とする。
- 危険操作には権限制御、確認、監査ログを必須とする。
