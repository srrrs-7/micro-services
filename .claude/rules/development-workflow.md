# Development Workflow

このリポジトリで非自明な変更を行う際の **作業手順** を規定する。生成コードの再生成、`go mod tidy`、ドキュメントのみの更新といった機械的変更を除き、以下は **必須** とする。

## 1. TDD ループ: plan → red → implement → green

すべての挙動変更は次のループで行うこと。

1. **Plan.** 変えたい観測可能な挙動を、入力と期待出力の形で書き出す（プランファイル、または `TaskCreate` のタスクとしてインライン化）。
2. **Red.** 新しい挙動を駆動するテストを追加し、実行する。実装前に必ず失敗することを確認する。
3. **Implement.** 失敗テストを通す最小コードを書く。
4. **Green.** 該当モジュールで `go test ./...` がすべてパスすることを確認してから次の挙動に進む。

必須事項:

- テストは実装と同じ変更内に、実装より **先に** 書くこと。実装が先で後からテストを書くのは TDD ではないため、本リポジトリのサービスコードでは禁止する。
- 機械的変更（`go mod tidy`、生成コードの再生成、ドキュメント更新）は TDD をスキップしてよい。スキップする場合は明示的に宣言する。
- "Red" を省略しないこと。実装前に通ってしまうテストは新しい挙動を検証していない。

各フェーズの成果物:

- **Plan**: テスト名と各テストが何をアサートするかの一覧。コードはまだ書かない。
- **Red**: 失敗テストがコミット（あるいはステージ）された状態。`go test ./...` を走らせ、想定通りに失敗していることを確認。
- **Implement**: 列挙したテストを通すための最小差分。ついでのリファクタは行わない。
- **Green**: 影響を受けたモジュールで `make test` がパスし、変更ファイルで `make lint` がパスする状態。

レイヤごとの「何をテストするか」は `.claude/rules/testing.md` を参照すること（`domain` / `service` / `route` / `infra`）。

## 2. Plan モード → Implement モード

複数ファイルまたは複数レイヤをまたぐ変更では、以下を **必須** とする。

1. **Plan モードを先に使う.** Claude Code の plan モード（または `TaskCreate` タスク）で、編集前に変更計画を起こす。計画には少なくとも次を列挙する: 触るファイル、編集順序、各ステップで追加するテスト。
2. **合意を取る.** 実装前にユーザーに計画を提示する。指摘があれば計画を直す。指摘されている間は編集に進まない。
3. **Implement.** 合意後にのみ編集する。計画の各項目は完了したらすぐにマークする。

必須事項:

- 複数ファイル変更で plan モードをスキップしないこと。30 秒の計画コストは 10 分の差し戻しコストより遥かに小さい。
- 単一ファイル・単一目的の編集（typo 修正、1 行設定変更）は plan モードをスキップしてよい。
- 実装中に計画の誤りに気づいたら、即興で進めず計画を更新する。計画と実際の差分はレビューに有用な成果物である。

## 3. Sub-agent の積極活用

Claude Code の sub-agent (`Agent` ツール + `subagent_type=...`) は first-class なツールとして扱う。タスクが定義済みエージェントに合致する場合、**必須** で利用する。

必須事項:

- コードベースの広域探索（grep / read を 3 回以上要するもの）は `Explore` エージェントに委譲する。メインループから直接探索しない。
- 進行中の Go コード変更レビューは `go-reviewer` エージェントに委譲する。
- 新規 HTTP エンドポイント追加は `endpoint-builder` エージェントに委譲する。
- スキーマ変更を伴わない新規 sqlc クエリ追加は `sqlc-query-author` エージェントに委譲する。
- スキーマ変更（DDL）は `migration-author` エージェントに委譲する。
- 互いに独立した複数の調査が必要な場合は、1 メッセージ内に複数の `Agent` ツール呼び出しを並列で発行する。

なぜ「推奨」ではなく「必須」か:

- Sub-agent はメインのコンテキストウィンドウを大きなファイル読み込みや検索結果から保護する。
- システムプロンプトがタスク特化されているため、リポジトリ事情を踏まえたフォーカスのある成果物が得られる。
- 並列実行により wall-clock 時間がスループットに変換される。

Sub-agent を使わない場面:

- 対象が既知のとき（既知パスは `Read`、特定シンボルは Bash 経由の `grep`）。
- エージェント起動コストを上回らない極小タスク（1 行修正など）。
- ユーザーが対話で作業を操作しており、思考過程を見せる必要がある場面。

検証:

- Sub-agent のサマリは「何をしようとしたか」を述べるもので、実際に何が起きたかを保証しない。書き込み系エージェントが「完了」を返した後、`git status` / `git diff` / 対象ファイルの `Read` で実際の変更内容を確認してからユーザーに完了報告すること。
