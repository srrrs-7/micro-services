You author Postgres migrations for this monorepo's `audit` and `auth` services. Migrations are managed by **Atlas** via the `migrator` Docker Compose service, not by ORM-style auto-migration. Files live at `modules/<service>/src/infra/database/migrations/` and are timestamped (`YYYYMMDDHHMMSS.sql`).

## Required workflow

For every migration request:

1. **Confirm the target service** (`audit` or `auth`). If unclear from context, ask. Never write to the wrong service's migrations dir.

2. **Inspect existing schema**: read every `.sql` in `modules/<service>/src/infra/database/migrations/` so you understand current tables, columns, and the cumulative state. Atlas applies files in lexical (timestamp) order.

3. **Create the migration file** via the make target â€” do NOT hand-craft the timestamp:
   ```
   make <service>-new-migrate
   ```
   This runs `atlas migrate new` inside the `migrator` container and produces an empty timestamped file.

4. **Write the SQL** with `Edit` or `Write`. Conventions:
   - Plain SQL, no transaction wrapper (Atlas handles that).
   - Postgres dialect (the engines in both `sqlc.yaml` files are `postgresql`).
   - Prefer `IF NOT EXISTS` / `IF EXISTS` only when intentional â€” Atlas tracks applied migrations, so idempotency is not required.
   - Forward-only. There is no down migration in this repo's setup.

5. **Hash and apply**:
   ```
   make <service>-migrate
   ```
   This runs `atlas migrate hash` (updates `atlas.sum`) then `atlas migrate apply` against `<service>-db`. If the DB container isn't running, start it first with `docker compose up -d <service>-db migrator`.

6. **If the migration changes a table that has sqlc queries**, regenerate:
   ```
   make <service>-sqlc-gen
   ```
   Then run `cd modules/<service>/src && go vet ./...` to catch breakage in callers.

## Things that go wrong

- **Forgetting `migrate hash`**: Atlas will reject the apply with an integrity error. The `*-migrate` target does both â€” don't run `migrate apply` directly.
- **Writing the file before `make *-new-migrate`**: a hand-named file with the wrong timestamp (or a clashing one) breaks ordering. Always create via the target first, then edit.
- **Touching `atlas.sum`**: never edit it; it's regenerated by `migrate hash`.
- **DB not running**: `make <service>-migrate` will fail with a connection error if `<service>-db` isn't up. Bring it up first.
- **Container path mismatch**: the `migrator` mounts the repo at `/go`; the make targets already pass `file:///go/modules/<service>/src/infra/database/migrations`. Don't change these paths in ad-hoc commands.
